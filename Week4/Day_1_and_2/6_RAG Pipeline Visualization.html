üìã Detailed Technical Reference

üìÑ Source File Formats

PDF (Portable Document Format)

Challenges: Complex layouts, tables, images, scanned documents
Text-based PDFs: Direct text extraction possible
Image-based PDFs: Require OCR (Optical Character Recognition)
Best for: Reports, research papers, manuals, books

DOCX/DOC (Microsoft Word)

Structure: XML-based format with styles and formatting
Advantages: Preserves structure, headings, lists
Metadata: Author, creation date, revision history
Best for: Business documents, proposals, documentation

HTML/Web Pages

Structure: DOM tree with semantic tags
Challenges: Navigation, ads, dynamic content
Techniques: Web scraping, API access, crawling
Best for: Knowledge bases, wikis, web articles

Markdown (.md)

Format: Plain text with markup syntax
Advantages: Easy to parse, human-readable
Common in: GitHub, documentation, README files
Best for: Technical documentation, notes

CSV/Excel (Tabular Data)

Structure: Rows and columns of data
Excel features: Multiple sheets, formulas, formatting
Processing: Pandas, openpyxl for structured extraction
Best for: Datasets, financial data, logs

JSON/XML (Structured Data)

JSON: Nested key-value pairs, API responses
XML: Hierarchical markup, configuration files
Advantages: Machine-readable, preserves structure
Best for: API data, configurations, structured content

‚úÇÔ∏è Chunking Strategies

Fixed-Size Chunking

Method: Split by character/token count
Chunk size: 256-512 tokens typical
Overlap: 10-20% recommended
Pros: Simple, predictable, fast
Cons: May break semantic units
Best for: Uniform documents, quick prototypes

Recursive Character Splitting

Method: Split by separators hierarchically
Separators: \n\n ‚Üí \n ‚Üí . ‚Üí space
Advantage: Preserves natural boundaries
LangChain: RecursiveCharacterTextSplitter
Best for: General-purpose text splitting

Sentence-Based Chunking

Method: Split by sentence boundaries
Tools: NLTK, spaCy, sentence-transformers
Grouping: Combine sentences to target size
Pros: Maintains semantic coherence
Best for: Question-answering systems

Semantic Chunking

Method: Use embeddings to find topic boundaries
Process: Embed sentences, cluster by similarity
Advantage: Respects content structure
Cost: More computationally expensive
Best for: Complex, diverse documents

Document-Structure Based

Method: Split by document elements
Elements: Sections, paragraphs, headings
Markdown: Split by headers (#, ##, ###)
HTML: Split by semantic tags
Best for: Structured documentation

Token-Based Chunking

Method: Split based on tokenizer output
Tokenizers: tiktoken (OpenAI), HuggingFace
Advantage: Precise control for model limits
Use case: Matching embedding model limits
Best for: Production systems with token limits

üí° Chunking Best Practices
  • Overlap: Add 10-20% overlap between chunks to avoid losing context at boundaries
  • Metadata: Include document title, section, page number with each chunk
  • Size optimization: Test different sizes (256, 512, 1024 tokens) for your use case
  • Preserve structure: Keep related sentences/paragraphs together
  • Avoid splitting: Don't break code blocks, tables, or lists

üéØ Retrieval & Re-ranking Techniques

Dense Retrieval (Vector Search)

Method: Semantic similarity via embeddings
Metrics: Cosine similarity, dot product, Euclidean
Algorithms: k-NN, ANN (HNSW, IVF)
Pros: Captures semantic meaning
Cons: May miss exact keywords
Best for: Conceptual, semantic queries

Sparse Retrieval (Keyword Search)

Methods: BM25, TF-IDF
BM25: Best-Match ranking function
Advantages: Exact matches, interpretable
Implementation: Elasticsearch, OpenSearch
Pros: Fast, handles rare terms well
Best for: Exact keyword matching

Hybrid Search

Combination: Dense + Sparse retrieval
Fusion: Reciprocal Rank Fusion (RRF)
Weights: Typically 0.7 semantic + 0.3 keyword
Implementation: Weaviate, Pinecone hybrid
Advantage: Best of both worlds
Best for: Production systems

Cross-Encoder Re-ranking

Models: ms-marco-MiniLM, bge-reranker
Process: Query + doc ‚Üí relevance score
Workflow: First retrieve top-K, then rerank
Advantage: Much more accurate than bi-encoders
Cost: Slower, expensive for large K
Best for: Final ranking of top candidates

Cohere Rerank

API: Rerank-english-v3.0, rerank-multilingual
Features: Production-ready, fast
Input: Query + up to 1000 documents
Output: Relevance scores + ranked list
Pricing: $2/1000 searches
Best for: Easy plug-and-play reranking

MMR (Maximal Marginal Relevance)

Purpose: Diversify results, reduce redundancy
Balance: Relevance vs diversity
Lambda: 0.5 = equal weight to both
Use case: Avoid similar chunks
Implementation: LangChain, LlamaIndex
Best for: Diverse information needs

Contextual Compression

Method: Extract only relevant parts of chunks
Tools: LLM-based extractors
Process: Retrieve ‚Üí compress ‚Üí feed to LLM
Advantage: Reduces token usage
Use case: Long documents, token limits
Best for: Optimizing context window

Query Expansion

Methods: Synonyms, related terms, LLM expansion
Multi-query: Generate multiple query variations
HyDE: Generate hypothetical document, embed it
Advantage: Better recall
Implementation: LangChain MultiQueryRetriever
Best for: Improving retrieval recall

üèÜ Recommended Retrieval Pipeline
  1. Initial Retrieval: Hybrid search (dense + sparse) ‚Üí get top 50-100 candidates
  2. Re-ranking: Cross-encoder or Cohere Rerank ‚Üí narrow to top 5-10
  3. Diversity: Optional MMR to avoid redundancy
  4. Compression: Extract only relevant sentences if context is large
  5. Generation: Feed curated context to LLM

üìä RAG Evaluation Metrics & Best Practices

üéØ Retrieval Quality

  • Recall@K: % of relevant docs in top K results
  • Precision@K: % of retrieved docs that are relevant
  • MRR: Mean Reciprocal Rank of first relevant doc
  • NDCG: Normalized Discounted Cumulative Gain

‚úÖ Generation Quality

  • Faithfulness: Answer grounded in context
  • Answer Relevance: Addresses the question
  • Context Relevance: Retrieved docs are useful
  • BLEU/ROUGE: Text similarity metrics

‚ö° Best Practices

  • Optimize chunk size (typically 256-512 tokens)
  • Add chunk overlap (10-20%)
  • Include metadata in embeddings
  • Implement re-ranking for better results
  • Use hybrid search (vector + keyword)
  • Monitor and evaluate continuously
